# サンプルコードREADMEの執筆ルール

`chapter*/README.md` および `appendix/README.md` を書く・更新する際のルール。

## 基本方針

**全章のREADMEは同じ構造で統一する。他章の型を真似るのは正しい。**

- READMEの役割は次の2つに絞る：
    1. **書籍本文に登場するコマンドを節見出しごとに並べて、読者がコピペしやすくする**（メイン機能）
    2. **サンプルコードを手元で試したい人向けの超簡潔ガイドを冒頭リード文に添える**（サブ機能）
- ごちゃごちゃさせない。動作確認の完全な手順書を書こうとしない（それは書籍本文の役割）
- **ハンズオン章（第4, 13, 15章）のREADMEでは、書籍で言及しないCLI生成ファイル（`mcp_client/`、`model/` など雛形のまま未使用のもの）について「これらはCLIが生成する雛形で本書では使用しません」と簡潔に注記してよい**。読者が不要に見えるファイルで迷わないようにするため

## 統一テンプレート

```
# 第N章 章タイトル

サンプルコードを実際に動かしてみたい方のために、書籍に掲載されているコマンドをコピペしやすい形で掲載しています。

（必要に応じて、動作前提を1〜2文で補足。例：「本章の各サンプルコードは、書籍本文の手順で〇〇を作成し、コード内のIDをご自身の環境の値に置き換えてから実行してください」）

## セットアップ（または「事前準備」）

依存関係をインストール

```bash
uv sync
```

## X.Y 【ラベル】節タイトル

### X.Y.Z 項見出し（書籍本文の項番号）

```bash
書籍本文に登場するコマンド
```
```

## 冒頭リード文のルール

- **1文目は原則「サンプルコードを実際に動かしてみたい方のために、書籍に掲載されているコマンドをコピペしやすい形で掲載しています。」で始める**
- 解説章（第7, 8, 14章など本文にコマンド操作がない章）では、冒頭で「本章は〇〇の解説章のため、書籍本文中にコマンド操作はありません。サンプルコードを実際に動かしてみたい方のために、以下は手元で動かす場合の補助手順を記載しています。」と書き出してよい
- 動作前提（プレースホルダー置換、AWSリソース事前作成、デプロイ前提、ナレッジベース必須など）がある章は、リード文の直後に1〜2文で補足する
- ハンズオンとサンプル単体実行が混在する章（第3, 5章）は、リード文で「本章は〇〇（ハンズオン）と〇〇（スニペット）の両方を扱います」と明示する

## 章タイプ別のリード文例

| 章タイプ | 対象章 | リード文の型 |
|---|---|---|
| ハンズオン章 | 第4, 13, 15章 | 「この章のハンズオンが実施しやすいように、書籍に掲載されているコマンドをコピペしやすい形で掲載しています。」 |
| ハンズオン+スニペット混在章 | 第3, 5章 | 基本文 ＋「本章はハンズオン（〇.〇節）とスニペット（〇.〇節）の両方を扱います。」 |
| 自己完結スニペット章 | 第1, 10章 | 基本文 ＋「各スニペット／サンプルは単体で実行できます。」（複数ファイル連携がある場合は追記） |
| 参照型スニペット章 | 第6, 7, 8, 12章 | 基本文 ＋「本章の各サンプルコードは、書籍本文の手順で〇〇を作成し、コード内のIDをご自身の環境の値に置き換えてから実行してください。」 |
| 差分章 | 第11章 | 基本文 ＋「本章のサンプルコードは、〇〇章のハンズオンで作成したプロジェクトに差し込む差分ファイル群です。」 |
| RAG応用章 | 第14章 | 解説章の書き出し ＋「〇〇章のハンズオンで作成したナレッジベースを利用します。〇〇章が未実施の場合は先に実施してください。」 |
| 環境構築（付録） | 付録 | 「本書のハンズオンを実施する際の事前準備として、書籍に掲載されているコマンドをコピペしやすい形で掲載しています。」＋ コマンドが含まれる範囲を1文補足 |

## 節レベル見出しと【ハンズオン】/【サンプルコードのみ】ラベル

各節レベル見出し（`## X.Y タイトル`）には、その節がハンズオンか単体実行用スニペットかを明示するラベルを付与する。読者が「この節は手を動かして手順を進めるパートか、単にサンプルが置いてあるパートか」を節単位で識別できるようにするため。

### ラベルの種類

| ラベル | 用途 | 該当例 |
|--------|------|--------|
| `【ハンズオン】` | 書籍本文中で読者がコマンドを順次実行する手順が含まれる節 | 第1章 1.5節、第4章全節、第5章 5.3節、第13章全節、第15章全節、付録 付録2 |
| `【サンプルコードのみ】` | 書籍本文ではコードを示すだけで実行コマンドはなく、READMEに掲載するのは「サンプルコードを単体で動かしたい人向けの補助手順」である節 | 第3章 3.2/3.3/3.6/3.7節、第5章 5.2節、第6〜12章の各節 |

### フォーマット

```
## X.Y 【ラベル】節タイトル
```

- 節番号 → 半角スペース → `【ラベル】` → 節タイトル の順（ラベルは見出し先頭ではなく節タイトルの直前）
- 節タイトルが原稿でもともと「【ハンズオン】」を含む場合（例：「5.3 【ハンズオン】AIエージェントを〜」）は、そのまま流用してOK（再追記不要）
- 章全体が単一カテゴリ（全節ハンズオン、全節サンプルコードのみ）の場合でも、各節にラベルを付与する。読者が節単位で識別できることを優先する

### 階層ルール

- `##` = 節レベル（X.Y）。**ラベル必須**
- `###` = 項レベル（X.Y.Z）。ラベル不要
- `####` = サブ項レベル（付録2.3 配下の「uvのインストール」など）。ラベル不要
- 「事前準備」「セットアップ」「ファイル一覧」など節番号を持たない補助セクションはラベル不要で `##` のまま使える

### 並び順

READMEに掲載する節は **書籍本文と同じ節順** で並べる。混在章でも「ハンズオン節をまとめて先頭に」のような並べ替えはしない。書籍と1対1で照合できる順序を優先する。

## コマンドブロックのルール

- **`uv sync` は独立したコードブロックとして置く**（他コマンドと圧縮しない）
- 節見出しは書籍本文の節番号（例：`## 6.2 【サンプルコードのみ】短期記憶`）または「事前準備」「セットアップ」「サンプルアプリの起動」など役割名
- 項見出しは `### 6.2.2 会話イベントの保存と取得` のように節タイトルそのまま（ラベル不要）
- **`（書籍本文には記載なし）` の注記は付けない**。本文節番号がついているセクションは本文コマンド、それ以外（「事前準備」など）は補助手順、と自然に区別できるため、明示的な注記は冗長
- 書籍本文に登場するコマンドと、手元で試すための補助コマンドが混在する場合、節見出しのラベルで役割を分ける

## 出版後のお知らせ・コマンド修正ルール

出版後にアップデート起因のエラー回避などでコピペ用コマンドを修正する場合は、ルートREADMEと該当章READMEをセットで更新する。

- ルートREADMEは `## 📣 新着のお知らせ（詳細は後述）` に短い要約を1行追加し、`## 📗 お知らせ詳細` に章別の詳細を書く
- 詳細は `### 第N章` 配下に `P.xxx：` 形式で書き、書籍の該当ページと修正済みの章READMEを明示する
- 該当章READMEでは、対象の `### X.Y.Z` 見出し直下に、引用ではないプレーンテキストで更新行を1行追加する
- 更新行は「【YYYY/M/D更新】CDKのアップデートに伴うエラー回避のため、AgentCore CLIのインストールコマンドを修正しました。」のように、理由と修正対象を簡潔に書く
- 書籍のmemoや章READMEのコマンドブロックに同じコマンドがある場合、ルートREADMEの詳細にはコマンドを重複掲載しない
- コマンドブロック本体は、読者がそのままコピペできる最新の内容に修正する

## 禁則事項（過去の失敗から）

- **動かないコマンドを動く風に書かない**。動作前提がある場合は冒頭リード文で言及する
- **コード種別と実行可否を区別する**。以下のファイルは「デプロイ／配置するもの」であり、`uv run` での実行コマンドをREADMEに書かない：
    - Lambda ハンドラーコード（`def lambda_handler` のみ定義）
    - Cedar ポリシーファイル（`.cedar`）
    - Dockerfile
    - pyproject.toml
    - CloudFormation / CDKテンプレート
- **AgentCoreランタイムへのデプロイ前提のコード**（`app.run()` 系）に対して、ローカルの `uv run` を「動作確認」として書かない
- **プレースホルダー（`<ここに...>`）を含むコード** に対して、置換前提の言及なしに `uv run` を書かない
- **節レベル見出しに `【ハンズオン】` / `【サンプルコードのみ】` ラベルを付け忘れない**。混在章で読者を混乱させる原因になる

## README執筆・更新時のチェック観点

- [ ] 冒頭リード文は定型句「サンプルコードを実際に動かしてみたい方のために、〜」で始まっているか？
- [ ] 動作前提（プレースホルダー置換、AWSリソース事前作成、デプロイ前提など）がある章で、リード文直後にそれを1〜2文で補足しているか？
- [ ] **節レベル見出し（`## X.Y タイトル`）に `【ハンズオン】` または `【サンプルコードのみ】` のラベルが付与されているか？**
- [ ] **項レベルは `###`（3つ）を使い、`##` を流用していないか？**
- [ ] 節の並び順は書籍本文と同じ順序になっているか？（混在章でも書籍順を維持）
- [ ] 書いた `uv run xxx.py` は、読者がプレースホルダー置換・AWSリソース事前作成をしていれば素朴に動くか？
- [ ] Lambda ハンドラー等の「デプロイ／配置するもの」に対して `uv run` を書いていないか？
- [ ] `uv sync` を独立コードブロックとして置いているか？（他コマンドと圧縮していないか）
- [ ] `（書籍本文には記載なし）` のような冗長な注記を付けていないか？
- [ ] 他章のREADMEと同じ構造（リード文 + 節見出しごとのコマンドブロック）になっているか？
- [ ] 出版後のコマンド修正では、ルートREADMEのお知らせ詳細と該当章READMEの見出し直下の更新行がセットになっているか？

## 過去の失敗例

### 2026-04-21 動かないコマンドが多数混入（authoring 側で発覚）

第11章のREADMEで、AgentCoreランタイムへのデプロイ前提のコード（`app.run()` + `os.environ[...]`）に対して `uv run main.py` を「動作確認」として書いた。環境変数未設定で `KeyError`、仮に設定してもHTTPサーバー起動だけでエージェント呼び出しに到達しない、という完全に動かないコマンドになっていた。

同日に全章のREADMEを再点検した結果、第6, 7, 8, 12, 14章で同様の問題（プレースホルダー置換必須、AWSリソース事前作成必須、Lambdaコードに対する `uv run` など）が発覚し、冒頭リード文で動作前提を1〜2文明示する形に修正した。同時に冒頭の定型句「サンプルコードを実際に動かしてみたい方のために、〜」を全章で統一した。

### 2026-05-10 ハンズオンとスニペットの混在で読者が混乱

第5章READMEで、ハンズオン手順（5.3節）とサンプル単体実行（5.2節）が同じ節レベル（`## 5.2.3` `## 5.3.2` ...）に並んでおり、読者が「どこからどこまでが手順を進めるパートか」を識別できなかった。同時に第5章 5.3.2 の `mkdir chapter5` / `cd chapter5` と第15章 15.6.2 の `echo $ACCOUNT_ID` の検算行が漏れていた。

対応として、節レベル見出しに `【ハンズオン】` / `【サンプルコードのみ】` ラベルを付与し、項レベル（X.Y.Z）を `###` に1段下げる構造に統一した。混在の有無にかかわらず全章で同じ構造に揃えることで、読者が節単位で「これから何を読むのか」を即座に判断できるようにした。
